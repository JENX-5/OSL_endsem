Section 1 — Foundation

What is Deadlock?

🧠 Real-World Analogy First

Imagine a one-lane bridge. Car A enters from the left. Car B enters from the right. Neither can reverse (they're holding their lane). Both are waiting for the other to move. Neither can ever proceed. That is deadlock.

In OS terms: Process P1 holds resource R1 and needs R2. Process P2 holds R2 and needs R1. Circular wait → permanent block.

Formal Definition

A set of processes is in deadlock if every process in the set is waiting for an event that can only be caused by another process in the same set. Since all are waiting, none can proceed — ever.

Process State in Deadlock

Every process is in the blocked/waiting state. No process is running or ready. CPU is idle even though work is queued.

Resources Involved

Can involve any resource: CPU cycles, memory, I/O devices, files, semaphores, locks — anything that can only be used by one process at a time.

Starvation vs Deadlock

Deadlock: A process will never get the resource. Starvation: A process might eventually get the resource but keeps getting skipped. Deadlock is worse — it is permanent.

Section 2 — Foundation

The 4 Necessary Conditions

Deadlock can occur if and only if all four of these hold simultaneously. Remove ANY ONE and deadlock is impossible. (These are Coffman's Conditions, 1971.)

1. Mutual Exclusion

At least one resource must be non-sharable. Only one process can use it at a time. Others must wait.

Example: A printer.

2. Hold and Wait

A process holds ≥1 resource AND is waiting to acquire more resources currently held by others.

Example: P1 holds R1, waits for R2.

3. No Preemption

Resources cannot be forcibly taken away. A process only releases a resource voluntarily when done with it.

Example: You can't grab a file mid-write.

4. Circular Wait

A set {P0, P1, …, Pn} where P0 waits for P1, P1 waits for P2, …, Pn waits for P0. A cycle exists.

Example: P1→R2→P2→R1→P1

⚡ Prevention Strategy

To prevent deadlock, you design the system so that at least one condition can never hold:

  • Break Mutual Exclusion: Use sharable resources (read-only files). Often impossible.
  • Break Hold-and-Wait: Require all resources at once before starting. Leads to low utilization.
  • Break No-Preemption: If a process can't get what it needs, release all it holds. Complex rollback.
  • Break Circular Wait: Assign total ordering to resources; processes must request in order. Most practical!
Section 3 — Foundation

Resource Allocation Graph (RAG)

The RAG is the visual tool used to detect and analyze deadlock. Understanding it is essential for exams.

Nodes

  • Process node (circle): Pi
  • Resource node (rectangle): Rj with dots = instances

Edges

  • Request edge: Pi → Rj (process wants resource)
  • Assignment edge: Rj → Pi (resource given to process)

The Critical Rule

Single Instance Resource Type

If a cycle exists in the RAG → DEADLOCK IS CERTAIN. No exceptions.

Multiple Instance Resource Type

Cycle exists → DEADLOCK POSSIBLE but not certain. Need Banker's algorithm to confirm.

Why? Intuition

With single instances, a cycle means A waits for B waits for A — literally no way out. With multiple instances, another copy of the resource might be freed by a process outside the cycle, breaking the wait.

Section 4 — Algorithms

Deadlock Avoidance

Instead of preventing deadlock by restricting resource usage, avoidance allows the OS to dynamically decide: "Should I grant this request right now, or will it lead to deadlock?"

🏦 The Bank Analogy

Think of the OS as a bank. Each process is a customer who declares their maximum loan (max resources needed). The bank only lends if it can still serve all customers even in the worst case. A state is "safe" if the bank can always satisfy everyone eventually — even if some customers need to wait.

Safe State

A system state is safe if there exists a safe sequence — an ordering of all processes such that for each process Pi, the resources Pi still needs can be satisfied by:

Available resources + Resources held by all Pj where j < i

Safe State → No Deadlock

System can always find an ordering to complete all processes. Even if some wait, they will eventually finish.

Unsafe State → Possible Deadlock

No safe sequence exists. Deadlock may or may not occur depending on future requests — but we're at risk.

⚠️ Key Insight

Not all unsafe states lead to deadlock. But the OS refuses to enter unsafe states as a precaution. This is conservative — resource utilization suffers, but deadlock is impossible.

Section 5 — Algorithms

Banker's Algorithm High Exam Weight

Designed by Dijkstra. Used when there are multiple instances of each resource type. It checks if granting a request keeps the system in a safe state.

Data Structures

Let n = number of processes, m = number of resource types.

StructureSizeMeaning
Available[m]mAvailable[j] = number of free instances of resource Rj
Max[n][m]n × mMax[i][j] = max instances of Rj that Pi may ever need
Allocation[n][m]n × mAllocation[i][j] = instances of Rj currently given to Pi
Need[n][m]n × mNeed[i][j] = more instances Pi still needs = Max[i][j] − Allocation[i][j]
Request[m]mThe current request vector from process Pi
Need[i][j] = Max[i][j] − Allocation[i][j]

Safety Algorithm

Determines if the current state is safe. Complexity: O(m × n²).

PSEUDOCODE 
// Safety Algorithm: returns true if system is in safe state
function isSafe(Available, Allocation, Need, n, m):
    Work[m]  ← Available           // work = copy of available
    Finish[n] ← {false, ..., false}  // no process finished yet
    
    loop:
        found ← false
        for i = 0 to n-1:
            if Finish[i] == false AND Need[i] ≤ Work:
                Work  ← Work + Allocation[i]  // Pi finishes, releases resources
                Finish[i] ← true
                found ← true
                record i in safe_sequence
        if not found: break
    
    if all Finish[i] == true: return SAFE + safe_sequence
    else: return UNSAFE
  1. Initialise: Work = Available. Finish[i] = false for all i.
  2. Find eligible process: Look for any i where Finish[i] = false AND Need[i] ≤ Work. (Process can complete with available resources.)
  3. Simulate completion: Set Work = Work + Allocation[i]. Set Finish[i] = true. Add i to safe sequence. Go back to step 2.
  4. Check result: If all Finish[i] = true → SAFE STATE. If any Finish[i] = false → UNSAFE STATE (those processes are deadlocked).

Resource-Request Algorithm

Called when process Pi makes a request. Only grants if system stays safe.

PSEUDOCODE 
function requestResources(i, Request):
    // Step 1: Request must not exceed declared need
    if Request[i] > Need[i]:
        error("Process exceeded maximum claim!")
    
    // Step 2: Resources must be available right now
    if Request[i] > Available:
        wait()  // resources not available, Pi must block
    
    // Step 3: Pretend to allocate, then check safety
    Available  ← Available  - Request[i]
    Allocation[i] ← Allocation[i] + Request[i]
    Need[i]    ← Need[i]    - Request[i]
    
    if isSafe():
        // Allocation is committed — Pi gets resources
        return GRANTED
    else:
        // Roll back — state was unsafe
        Available  ← Available  + Request[i]
        Allocation[i] ← Allocation[i] - Request[i]
        Need[i]    ← Need[i]    + Request[i]
        wait()

Section 8 — Algorithms

Deadlock Detection Algorithm

Unlike avoidance (which prevents unsafe states), detection allows deadlocks to happen and then detects them. Used when overhead of avoidance is too high.

Data Structures

  • Available[m]: free resource instances
  • Allocation[n][m]: currently allocated
  • Request[n][m]: current outstanding requests
  • (No Max / Need — we don't know maximum demands)

When to Invoke?

  • Every time a request cannot be granted immediately
  • Periodically (every hour, every N allocations)
  • When CPU utilization drops below threshold

Detection Algorithm

Very similar to Safety Algorithm, but uses Request instead of Need.

PSEUDOCODE 
function detectDeadlock(Available, Allocation, Request, n, m):
    Work[m]  ← Available
    for i = 0 to n-1:
        // Processes with no allocations are trivially "finished"
        Finish[i] ← (Allocation[i] == 0) ? true : false
    
    loop:
        found ← false
        for i = 0 to n-1:
            if Finish[i] == false AND Request[i] ≤ Work:
                Work ← Work + Allocation[i]
                Finish[i] ← true
                found ← true
        if not found: break
    
    deadlocked = {i | Finish[i] == false}
    if deadlocked is empty: return NO_DEADLOCK
    else: return DEADLOCK, deadlocked

Section 9 — Algorithms

Deadlock Recovery

Once deadlock is detected, the system must recover. Two main approaches:

1. Process Termination

  • Abort all deadlocked processes: Drastic, immediate. All work lost.
  • Abort one at a time: Kill cheapest process, re-run detection, repeat until no deadlock. Less loss, more overhead.

2. Resource Preemption

  • Selecting a victim: Which process/resource to preempt? Minimize cost.
  • Rollback: Roll process back to a safe state and restart it.
  • Starvation risk: Same process always picked as victim? Must limit rollback count.

Victim Selection Criteria

  • Process priority (kill lowest priority first)
  • How long the process has been running
  • Resources used / resources still needed
  • Number of processes that need to be terminated
  • Is the process interactive or batch?
Section 10 — Implementation

C Code Implementation

C 
/*
 * Banker's Algorithm for Deadlock Avoidance
 * Implements Safety Check + Resource-Request Algorithm
 * Matches Lab Manual Example: 5 processes, 3 resource types (A,B,C)
 */

#include <stdio.h>
#include <stdbool.h>

#define MAX_P 10   /* max processes */
#define MAX_R 10   /* max resource types */

int n, m;                          /* n=processes, m=resource types */
int available[MAX_R];
int maximum[MAX_P][MAX_R];
int allocation[MAX_P][MAX_R];
int need[MAX_P][MAX_R];

/* Compute Need = Max - Allocation for all processes */
void computeNeed() {
    for (int i = 0; i < n; i++)
        for (int j = 0; j < m; j++)
            need[i][j] = maximum[i][j] - allocation[i][j];
}

/* Check if vector a <= vector b (component-wise) */
bool vecLeq(int a[], int b[]) {
    for (int j = 0; j < m; j++)
        if (a[j] > b[j]) return false;
    return true;
}

/* Safety Algorithm: returns true if safe, fills safeSeq[] */
bool isSafe(int safeSeq[]) {
    int  work[MAX_R];
    bool finish[MAX_P] = {false};

    /* Step 1: Initialize Work = Available */
    for (int j = 0; j < m; j++)
        work[j] = available[j];

    int count = 0;

    /* Step 2 & 3: Find eligible processes and simulate */
    while (count < n) {
        bool found = false;
        for (int i = 0; i < n; i++) {
            if (!finish[i] && vecLeq(need[i], work)) {
                /* Pi can complete — simulate it releasing resources */
                for (int j = 0; j < m; j++)
                    work[j] += allocation[i][j];  /* release Pi's resources */
                finish[i] = true;
                safeSeq[count++] = i;
                found = true;
            }
        }
        if (!found) break;  /* no eligible process found — unsafe */
    }

    /* Step 4: All finished = safe state */
    return count == n;
}

/* Resource-Request Algorithm for process pid */
void requestResources(int pid, int request[]) {
    printf("\n--- Request from P%d ---\n", pid);

    /* Check 1: Request must not exceed declared need */
    if (!vecLeq(request, need[pid])) {
        printf("ERROR: Request exceeds maximum claim!\n");
        return;
    }

    /* Check 2: Resources must be available now */
    if (!vecLeq(request, available)) {
        printf("Wait: Resources unavailable. P%d must block.\n", pid);
        return;
    }

    /* Check 3: Pretend to allocate and test safety */
    for (int j = 0; j < m; j++) {
        available[j]       -= request[j];
        allocation[pid][j] += request[j];
        need[pid][j]       -= request[j];
    }

    int seq[MAX_P];
    if (isSafe(seq)) {
        printf("GRANTED! New Available: ");
        for (int j = 0; j < m; j++)
            printf("%d ", available[j]);
        printf("\nSafe sequence: <");
        for (int i = 0; i < n; i++)
            printf("P%d%s", seq[i], i==n-1?"":",");
        printf(">\n");
    } else {
        /* Roll back — would make system unsafe */
        printf("DENIED: Would cause unsafe state. Rolling back.\n");
        for (int j = 0; j < m; j++) {
            available[j]       += request[j];
            allocation[pid][j] -= request[j];
            need[pid][j]       += request[j];
        }
    }
}

void printMatrices() {
    printf("\n%-8s%-18s%-18s%-18s\n",
        "Proc", "Allocation", "Max", "Need");
    for (int i = 0; i < n; i++) {
        printf("P%-7d", i);
        for (int j = 0; j < m; j++) printf("%d ", allocation[i][j]);
        printf("         ");
        for (int j = 0; j < m; j++) printf("%d ", maximum[i][j]);
        printf("         ");
        for (int j = 0; j < m; j++) printf("%d ", need[i][j]);
        printf("\n");
    }
    printf("Available: ");
    for (int j = 0; j < m; j++) printf("%d ", available[j]);
    printf("\n");
}

int main() {
    /* Lab manual example */
    n = 5; m = 3;

    int alloc[5][3] = {{0,1,0},{2,0,0},{3,0,2},{2,1,1},{0,0,2}};
    int max_r[5][3] = {{7,5,3},{3,2,2},{9,0,2},{2,2,2},{4,3,3}};
    int avail[3]    = {3,3,2};

    for (int i=0; i<n; i++)
        for (int j=0; j<m; j++) {
            allocation[i][j] = alloc[i][j];
            maximum[i][j]    = max_r[i][j];
            available[j]     = avail[j];
        }

    computeNeed();
    printMatrices();

    int safeSeq[MAX_P];
    if (isSafe(safeSeq)) {
        printf("\nSystem is in SAFE STATE\nSafe Sequence: <");
        for (int i=0; i<n; i++)
            printf("P%d%s", safeSeq[i], i==n-1?"":",");
        printf(">\n");
    } else {
        printf("\nSystem is in UNSAFE STATE!\n");
    }

    /* Test request from P1: (1,0,2) */
    int req1[] = {1,0,2};
    requestResources(1, req1);

    /* Test request from P4: (3,3,0) */
    int req4[] = {3,3,0};
    requestResources(4, req4);

    return 0;
}

/*
 * Deadlock Detection Algorithm
 * No Max/Need matrices — processes make ad-hoc requests
 * Detects which processes are deadlocked
 */

#include <stdio.h>
#include <stdbool.h>
#include <string.h>

#define MAX_P 10
#define MAX_R 10

int n, m;
int available[MAX_R];
int allocation[MAX_P][MAX_R];
int request_mat[MAX_P][MAX_R];  /* current outstanding requests */

bool vecLeq(int a[], int b[]) {
    for (int j = 0; j < m; j++)
        if (a[j] > b[j]) return false;
    return true;
}

bool isZeroRow(int row[]) {
    for (int j = 0; j < m; j++)
        if (row[j] != 0) return false;
    return true;
}

void detectDeadlock() {
    int  work[MAX_R];
    bool finish[MAX_P];

    /* Init: work = available */
    for (int j = 0; j < m; j++)
        work[j] = available[j];

    /* Init finish[i]: true if no allocation (trivially done) */
    for (int i = 0; i < n; i++)
        finish[i] = isZeroRow(allocation[i]);

    /* Find completable processes */
    bool changed = true;
    while (changed) {
        changed = false;
        for (int i = 0; i < n; i++) {
            if (!finish[i] && vecLeq(request_mat[i], work)) {
                for (int j = 0; j < m; j++)
                    work[j] += allocation[i][j];
                finish[i] = true;
                changed = true;
                printf("  P%d can complete (Request ≤ Work)\n", i);
            }
        }
    }

    /* Report results */
    bool deadlock = false;
    printf("\n=== DEADLOCK DETECTION RESULT ===\n");
    for (int i = 0; i < n; i++) {
        if (!finish[i]) {
            printf("DEADLOCKED: P%d\n", i);
            deadlock = true;
        }
    }
    if (!deadlock) printf("No deadlock detected.\n");
}

int main() {
    n = 5; m = 3;

    /* Lab manual Example 2 data */
    int alloc[5][3]   = {{0,1,0},{2,0,0},{3,0,3},{2,1,1},{0,0,2}};
    int req[5][3]     = {{0,0,0},{2,0,2},{0,0,0},{1,0,0},{0,0,2}};
    int avail[3]      = {0,0,0};

    for (int i=0; i<n; i++)
        for (int j=0; j<m; j++) {
            allocation[i][j]  = alloc[i][j];
            request_mat[i][j] = req[i][j];
        }
    for (int j=0; j<m; j++)
        available[j] = avail[j];

    printf("Running detection...\n");
    detectDeadlock();

    /* Part (b): P2 requests 1 more C */
    printf("\n--- P2 requests 1 more C ---\n");
    request_mat[2][2] = 1;
    printf("Re-running detection...\n");
    detectDeadlock();

    return 0;
}
Section 11 — Exam Prep

Common Mistakes

✗
Confusing Avoidance vs Detection
Avoidance prevents unsafe states BEFORE they happen. Detection finds deadlocks AFTER they occur. Many students mix up which uses Need vs Request matrices.
✗
Forgetting the "Pretend" Step in Request Algorithm
You must actually modify Available, Allocation, Need — then run safety check. If unsafe, ROLL BACK all three. Students forget the rollback.
✗
Need Matrix in Detection Algorithm
Detection does NOT use Need or Max. It uses the actual outstanding Request matrix. Detection algo uses Request ≤ Work (not Need ≤ Work).
✓
Safe sequence is not unique
Multiple safe sequences may exist. Any valid one is acceptable as the answer. P1,P3,P4,P0,P2 and P3,P1,P2,P4,P0 can both be correct.
✗
Cycle in RAG = Deadlock (Wrong!)
Only true for single-instance resources. With multiple instances, a cycle is necessary but not sufficient for deadlock. Use detection algorithm to confirm.
✗
Available computed wrong
Available = Total − Σ(Allocation for all processes). Students often forget to subtract ALL allocations, or use Max instead of Allocation.
Section 12 — Exam Prep

Viva / Exam Questions

Short Answer

What are the four necessary conditions for deadlock? ▾
Mutual Exclusion, Hold and Wait, No Preemption, and Circular Wait. ALL four must hold simultaneously for deadlock to occur.
What is the difference between a safe and unsafe state? ▾
A safe state guarantees there exists at least one safe sequence — an order of processes that can all complete without deadlock. An unsafe state means no such sequence exists, meaning deadlock is possible (though not guaranteed) based on future requests.
What is Need[i][j] in Banker's Algorithm? ▾
Need[i][j] = Max[i][j] - Allocation[i][j]. It represents how many more instances of resource Rj process Pi may still request before it completes.
Why can't Banker's Algorithm be used in practice easily? ▾
It requires processes to declare their maximum resource needs IN ADVANCE. In practice, processes rarely know this ahead of time. Also, the number of processes and resources must be fixed — both change dynamically in real systems.
What is the time complexity of the Safety Algorithm? ▾
O(m × n²) — for each of the n processes, we may need to scan all n processes in the worst case, and comparing vectors takes O(m).

Long Answer / Trick Questions

🎯 TRICK: Can an unsafe state lead to no deadlock? ▾
YES! This is a key trick question. An unsafe state means deadlock is POSSIBLE, not certain. If processes happen to release resources in the right order, deadlock may never occur. The OS avoids unsafe states as a conservative precaution. The implication is one way: Safe → No deadlock. But Unsafe → MAYBE deadlock.
🎯 TRICK: Is deadlock avoidance better than deadlock prevention? ▾
Not always! Avoidance (Banker's) gives better resource utilization than many prevention strategies but requires advance knowledge of max demands. Prevention via resource ordering (break circular wait) is simpler and has no runtime overhead. The "better" choice depends on context — real-time systems prefer prevention; database systems may prefer detection+recovery.
🎯 Explain why Windows/Linux don't use Banker's Algorithm ▾
Three reasons: (1) Requires all processes to declare max resources upfront — impractical. (2) Number of processes and resources must be fixed and known — impossible in general-purpose OSes. (3) Runtime overhead of checking safety on every request. Instead, modern OSes use a combination of careful API design (avoiding circular wait by convention), watchdog timers, and deadlock detection in specific subsystems (like DB engines).
🎯 TRICK: Two processes, each needs resource held by the other. Is this always deadlock? ▾
Only if No Preemption holds. If the OS can preempt a resource (take it away and roll back), there's no deadlock. Also if one process finishes before the other starts needing the resource — timing matters. Deadlock requires ALL four Coffman conditions simultaneously.
How does Detection Algorithm differ from Safety Algorithm? ▾
Three differences: (1) Uses Request matrix (actual outstanding requests) instead of Need matrix. (2) Initializes Finish[i]=true if Allocation[i]=0 (not waiting at all), vs Safety which starts all at false. (3) If Finish[i]=false at end, those processes ARE deadlocked (not just "unsafe") — detection gives a definitive list of deadlocked processes.
Section 13 — Quick Revision

Summary Sheet

4 Conditions (ALL needed)

  • Mutual Exclusion
  • Hold & Wait
  • No Preemption
  • Circular Wait ← easiest to break

Key Formula

  • Need = Max − Allocation
  • Safe: Need ≤ Work (use Work+Alloc)
  • Detect: Request ≤ Work
  • Available = Total − ΣAllocation

Banker's Algorithm Steps

  • 1. Request ≤ Need? (else error)
  • 2. Request ≤ Available? (else wait)
  • 3. Pretend to allocate
  • 4. Run Safety check
  • 5. Safe→grant / Unsafe→rollback

Safety Algorithm Steps

  • 1. Work=Available, Finish=false
  • 2. Find i: !Finish[i] & Need≤Work
  • 3. Work+=Allocation[i], Finish[i]=T
  • 4. Repeat until none found
  • 5. All T → Safe; any F → Unsafe

RAG Rules

  • → Request edge (process wants)
  • ← Assignment edge (resource held)
  • Cycle + single instance = Deadlock
  • Cycle + multi instance = Maybe

Recovery Methods

  • Abort all deadlocked (drastic)
  • Abort one-by-one (expensive)
  • Preempt resources + rollback
  • Avoid starvation in victim selection

Complexities

  • Safety Algo: O(m × n²)
  • Detection Algo: O(m × n²)
  • Request check: O(m × n²) worst

Safe vs Unsafe

  • Safe → No deadlock (guaranteed)
  • Unsafe → Deadlock POSSIBLE
  • Deadlock → Always unsafe
  • Unsafe ≠ Deadlock (not always)
